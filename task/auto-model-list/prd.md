# PRD：模型列表自动拉取

## 问题

- 当前添加模型时，用户需逐个手动填写每个模型的 ID、名称等全部字段
- 对拥有数十个模型的供应商（OpenAI、DeepSeek、各类中转站），手动添加成本高、易出错、容易遗漏
- 大多数 OpenAI 兼容的中转站和 API 服务都提供 `GET /v1/models` 端点可获取模型列表，但当前产品未利用该能力
- 自定义供应商同样面临手动添加的负担，且中转站用户比例不低

## 目标与非目标

### 目标

- 为每个已配置 baseUrl 和 API key 的供应商（包括内置和自定义）提供「拉取模型列表」功能
- 调用 `GET {baseUrl}/models` 端点获取模型 ID，展示给用户选择导入
- 拉取失败时（网络错误、鉴权错误、不支持的 API 类型等），保留手动添加作为兜底
- 拉取成功时仅填充模型 `id` 字段，`name` 从 API 响应的 `id` 自动复制（或从 `name` 字段映射），其余字段保持默认值
- 覆盖所有 `openai-completions` 和 `openai-responses` API 类型的供应商（内置和自定义均支持）

### 非目标

- 不调用除模型列表 API 之外的任何网络请求
- 不预填模型的 `reasoning`、`input`、`contextWindow`、`maxTokens`、`cost` 等字段——仅填充 `id` 和 `name`，其余保持默认
- 不自动持续同步或定期刷新模型列表（每次用户主动触发）
- 不支持 `anthropic-messages` 和 `google-generative-ai` API 类型的模型列表拉取（这两类无标准的 list models 端点），仅支持手动
- 不修改现有模型验证规则、序列化规则或激活写入逻辑
- 不修改现有供应商 CRUD 逻辑

## 范围内外

### 范围之内

- 在方案编辑器的模型管理区域，为每个供应商增加「拉取模型列表」交互入口（按钮/操作）
- 后端新增一个 Wails API 方法 `FetchProviderModels(schemeID, providerKey)`，执行 HTTP GET 请求到 `{provider.BaseURL}/models`
- 请求携带 `Authorization: Bearer {provider.APIKey}` 头
- 解析 OpenAI 兼容的 List Models 响应格式：`{ "data": [ { "id": "...", "name": "...", ... } ] }`
- 返回解析后的模型 ID 列表给前端，前端展示供用户多选/全选后批量导入
- 导入时仅填充 `id` 和 `name`（`name` 优先取响应中 `name` 字段，无则复制 `id`）
- 拉取失败时（HTTP 非 200、网络不可达、超时、JSON 解析错误等），返回错误信息；前端展示错误提示并保持手动添加入口可用
- 已存在的模型 ID 重复时跳过导入

### 范围之外

- 不修改供应商 CRUD、方案 CRUD、激活写入等现有功能
- 不修改 `models.json` 序列化格式
- 不修改模型验证规则
- 不提供模型列表缓存（每次用户主动拉取都是实时请求）

### 网络调用豁免声明

本项目 AGENTS.md 中有「无网络调用：不校验 API key 或 endpoint，不发起 HTTP 请求」的硬规则。本功能是该项目规则在 v1 阶段设立后首个明确需要网络调用的需求。本 PRD 标记该规则在此功能范围内被豁免——调用 `GET /v1/models` 获取模型列表是用户主动触发的操作，不是校验或验证行为。

## 验收标准

### 正常场景

- **AC-01**：在方案编辑器中，为一个已配好 baseUrl 和 apiKey 的 `openai-completions` 类型供应商（内置或自定义），界面提供「拉取模型列表」按钮
- **AC-02**：点击按钮后，前端调用后端 `FetchProviderModels`，后端向 `{baseUrl}/models` 发送 GET 请求携带 `Authorization: Bearer {apiKey}` 头
- **AC-03**：请求成功返回 JSON `{ "data": [...] }` 后，弹窗展示模型列表（每项显示 `id` 和 `name`），支持多选/全选
- **AC-04**：用户勾选模型确认导入后，选中模型作为新条目添加到该供应商模型列表中；`id` 从响应 `id` 字段填充，`name` 优先取响应 `name`、无则复制 `id`
- **AC-05**：导入后用户可正常编辑每个模型的其他字段（contextWindow、maxTokens、cost 等），并可正常保存
- **AC-06**：对自定义供应商（`openai-completions` 类型，如 Ollama、中转站）执行 AC-01~AC-05，同样通过

### 拉取失败 / 兜底

- **AC-07**：拉取时网络不可达（DNS 解析失败、连接超时等），界面展示错误消息，用户仍可手动逐条添加模型
- **AC-08**：拉取时 API 返回非 200（401 鉴权失败、404 无 models 端点、500 等），界面展示具体 HTTP 状态码和错误消息，用户仍可手动添加
- **AC-09**：拉取时返回的 JSON 格式不符合预期（无 `data` 字段、非数组等），界面提示解析失败，用户仍可手动添加

### 重复与冲突

- **AC-10**：若选中的模型 ID 与供应商下已有模型 ID 重复，该模型被跳过不导入，不影响其余模型导入；界面提示跳过的模型数量
- **AC-11**：若选中的所有模型 ID 均已存在，导入操作不创建任何新模型，界面提示无新增

### API 类型限制

- **AC-12**：对 `anthropic-messages` 类型的供应商，「拉取模型列表」按钮不显示或置灰并提示"该 API 类型不支持自动拉取"
- **AC-13**：对 `google-generative-ai` 类型的供应商，同 AC-12

### 边界与错误

- **AC-14**：供应商无 baseUrl（如刚创建未填写的自定义供应商），「拉取模型列表」按钮置灰或隐藏，提示"请先配置 baseUrl"
- **AC-15**：供应商无 apiKey，「拉取模型列表」按钮可用（因为有些本地服务不需要 key），但请求时发空 Authorization 头
- **AC-16**：选中零个模型点击确认导入，操作不执行，界面提示"请至少选择一个模型"
- **AC-17**：未保存的模型编辑内容在拉取操作后不受影响；导入仅新增条目，不覆盖或删除已有条目
