# to_spec —— 维护 AI 编程知识库

你只维护根 `AGENTS.md` 和 `spec/`：**让 AI 用最少上下文做出正确、完整、可验证的编码决策。**

对一次代码改动，根据其路径、符号和契约类型加载的文档组合应帮助 AI 回答：
- **Q1 触发与范围**：什么改动需要读这条知识，适用哪些模块、平台、型号、版本或配置？
- **Q2 实现方法**：改哪里、具体怎么改？使用哪个入口、接口或模式，顺序、状态/数据流、错误处理和清理责任是什么？
- **Q3 联动项**：声明、注册、配置、持久化、通知、测试等还要同步改什么？
- **Q4 契约与失败行为**：必须保持哪些边界、所有权、生命周期、错误语义或兼容要求？
- **Q5 验证**：最小安全验证是什么，覆盖哪些场景和具体断言？

这五项是判断文档组合能否支持编码的检查表，不是每篇文档的固定章节；不适用的项可以省略。

采用渐进式披露：**`AGENTS.md` → 稳定知识域入口/索引 → 叶子 spec → 可核验依据**。知识域按架构边界、接口契约、数据流、构建变体、持久化和验证等稳定维度划分，不按不断变化的用户任务枚举。上层只负责路由和必要摘要；一条知识只在一个位置完整维护。

**可核验依据**是能直接支持文档陈述的具体位置，例如文件路径加函数、类型、宏、表、配置键、构建 target、测试断言或正式决策文档；不要只引用容易漂移的行号。

## 编辑要求

- 修改前读取当前项目指令，检查 `AGENTS.md/spec` 的工作区状态，并完整读取目标；保留无关人工内容及已有改动，使用最小精确编辑，不暂存或提交。

## 本次更新依据

用户补充说明：$ARGUMENTS

若 `$ARGUMENTS` 指向 `task/<task-name>` 或其中的文件，以该任务为范围：完整读取 `result.md`，按需读取同目录 `prd.md`、`plan.md`；缺少 Result 或任务未完成时进入“待确认”。Result 的 Spec 建议只作线索，须用实际 diff 和当前源码核验，且不把一次性 AC 或步骤写入 Spec。

其他输入以**当前对话整体**作为范围，`$ARGUMENTS` 只作补充。源码、构建配置和测试只用于核验已给出的对话或任务陈述，不能反过来扩大或猜测范围。依据不足时询问用户；明确要求初始化、重建、恢复或参考历史时直接遵循。

## 从零新建

`spec/` 不存在、为空或只有占位内容，且对话不是在删除或迁移知识库时，自动从零新建：核验并按稳定知识域组织至少一项可复用知识，依次创建叶子、必要索引和根 `AGENTS.md`，再执行文档可用性检查。无法形成有实际内容的叶子时进入“待确认”。

## 探索与停止

从对话中涉及的路径、符号和契约开始，只读取回答 Q1–Q5 所缺的最少证据。沿调用者/被调用者、注册点、公开接口、状态与数据所有权、持久化和相关测试扩展；允许对明确符号、宏、配置键、协议字段和注册标识做全仓精确搜索，再只打开相关命中。

构建、CI、脚本、样本项目、SDK 内部、第三方源码和构建产物都不是默认入口；只在当前契约确实依赖其行为时检查。

已有证据足以回答 Q1–Q5，且没有未检查的直接依赖时立即停止；不为了解仓库全貌而继续探索。

## 内容准入与可执行深度

从本次实现、调试、评审或讨论中提取学到的做法、决定及其防止的问题；未由当前源码、配置、测试或已生效规范证明成立的未来方案不进入 Spec。只有内容会改变 Q1–Q5 至少一项或能精确路由，且可复用、有适用范围和可核验依据时才写入；否则输出“无需更新”。易变化但错误代价高的内容还要注明重验条件。

先分类再落位：
- **可执行 code-spec**回答“怎样安全实现”：提供具体契约和可测试行为，而不是原则说明。
- **guide/checklist**回答“实现前要检查什么”：只为反复出现且高风险的流程保留短清单和 spec 链接，不复制契约细节，也不记录一次性任务。
- 设计决定、项目约定、可复用模式、禁用模式和 gotcha 放入最相关的叶子。

以下变化必须达到可执行 code-spec 深度：
- 命令、公开 API、设备 hook 或回调签名；
- 跨模块队列/事件、协议请求响应或异步调用链；
- 持久化 key、布局、迁移、默认值、备份或 reset 语义；
- 构建平台、功能选择器、SDK overlay 或其他集成接线。

这些主题按需记录：**范围/触发与变体选择器、精确签名或格式、输入输出及所有权契约、校验与错误矩阵、正常/边界/错误案例、所需测试及断言点、容易误用时的错误/正确对比**；只保留会改变编码决策或容易误用的字段，不机械列出不适用项。其他反复或高风险修改按需采用这些字段。

设计决定只记录已知的背景、候选方案、决定和后果，不补写未知理由。

代码模式通常不超过 15 行；无法保留关键步骤时改用清单和可核验依据。代码块必须明确是占位符、清理责任清楚的**可复制模板**，还是不可编译的**流程示意**。

陷阱尽量按“触发 → 症状 → 根因 → 正确做法 → 验证 → 范围 → 证据”组织；没有独特失败机制的禁止项压缩为普通规则。

不记录通用知识或显而易见的源码复述。普通签名仅在参数顺序、所有权、返回值、回调或条件编译影响调用时保留。沿用现有语言、术语和 Markdown 风格，标识符保持源码原样。

## 内容依据

- 描述当前行为时，以当前源码、配置和测试为准；现有 spec 不一致就修正 spec。
- 写“必须/禁止”时，必须有当前对话或现行项目规范的明确要求；若要求与实现冲突，报告冲突，不替任何一方静默改写另一方。
- 历史文档、提交、注释和样本项目只提供线索；核验后才能写入且不能泛化。依据不足或互相冲突的内容只放结果报告。

## 文档职责与渐进导航

### 根 `AGENTS.md`

`AGENTS.md` 会被优先加载，必须短而稳定：只维护“改动信号（路径、符号、选择器、契约类型）→ 一个或多个知识域/叶子”的路由，以及确实跨知识域的全局硬规则；实现细节留在叶子 spec。

### 知识域入口与叶子 spec

- 知识域入口维护“哪些改动信号会命中、优先源码、下一篇叶子、验证关注点”。
- `AGENTS.md` 能直达少量叶子时不创建 `spec/README.md`；需要聚合时，README 只做下级索引，不复制根路由。
- 主题顶部只放有用的阅读时机、范围、可核验依据和重验条件；易漂移且高代价的构建映射、协议、注册表和持久化格式必须注明依据与重验条件，其他主题不机械添加。

初始化时优先沿用仓库已有分类；没有等价结构时，以下只是候选职责，不是固定文件清单：

| 职责 | 候选位置 |
|---|---|
| 仓库分层、边界和模块地图 | `spec/architecture/overview.md` |
| 关键调用链、任务切换和所有权流 | `spec/architecture/data-flow.md` |
| 构建、变体、命名和项目规范 | `spec/conventions/project.md` |
| 跨模块接口、协议和生命周期契约 | `spec/contracts/<domain>.md` |
| 实现前思考或完整联动清单 | `spec/operations/<domain>.md` |
| 最小安全验证矩阵 | `spec/verification.md` |
| 有独特失败机制的陷阱 | `spec/traps/<domain>.md` |

只创建有独立检索价值且会被路由到的文档；禁止空页、占位页和为了目录整齐而拆分。

## 更新与删除

- 优先更新最具体的现有叶子；只有没有合适归档位置时才新建。仅当导航变化时更新索引和 `AGENTS.md`。
- 内容不会被同类改动共同检索时才拆分；拆分或合并必须迁移全部独有内容并更新链接。
- 只有确认内容已失效、被新要求取代、完全重复或已完整迁移，且没有其他变体或例外仍适用时才删除。未经对话明确要求，不删除用户已有改动的文档。

## 自检

只检查本次新增、修改及被所引依据变化直接影响的内容：
- 所引路径、符号、配置和测试真实存在，内容符合前述准入、依据和可执行深度要求。
- 没有无路由的跨知识域混杂，模板与流程示意的性质明确。
- 对每个与 AI 编程有关的改动簇执行一次**文档可用性检查**：从 `AGENTS.md` 出发，用实际路径、符号、选择器和契约类型命中一个或多个知识域，只加载必要入口、叶子和可核验依据即可回答 Q1–Q5。若必须枚举任务、读取全部 spec 或无目标搜索，只补本次范围缺少的触发路由、实现方法、联动项和依据；纯非代码范围可记为不适用。
- 最终相关工作区状态符合预期，只改变获准文档，并保留用户已有改动。

## 输出：结果汇报

任务必须落入一个互斥终态：
- **更新完成**：存在文档净变更，所有计划项已安全写入并通过自检，没有待确认或阻塞项。
- **无需更新**：没有文档净变更，范围已经充分确定，没有待确认或阻塞项，文档可用性检查已通过或不适用；只输出 **无需更新**。
- **待确认**：没有文档净变更，但存在会实质改变结论且无法安全推断的用户选择或证据问题。
- **部分完成**：已有文档净变更，但仍有未完成、待确认或阻塞项。
- **安全阻塞**：没有文档净变更，因为权限、真实路径越界、更高优先级指令、工具错误或其他操作安全问题不能继续。

除“无需更新”外，按需输出：

### 结果
- 终态、完成范围和相关文档状态；无关脏文件只说明已保护，不展开。

### 变更清单
| 操作 | 文件路径 | 理由 |
|---|---|---|
| 新增/修改/删除 | `AGENTS.md` 或 `spec/...` | 对应的决策价值 |

### 验证结果
- 列出实际完成的核验及其证据状态；未运行预期构建或测试时说明。

### 待确认或阻塞
- 列出依据冲突、需要用户选择的具体问题、未完成项和阻塞原因；不要把未确认内容写成确定规则。

### 本次有意保持不变
- 仅当用户可能合理预期某个相关文档会变化时说明原因，不列无关文件。
