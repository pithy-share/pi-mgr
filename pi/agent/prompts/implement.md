要求：

- 必须使用参数指定的唯一 `task/<task-name>/plan.md` 及同目录 `prd.md`；当前对话只能补充说明，不能代替落盘文件。路径缺失、无效、两者冲突，或存在会改变实现和验收的未解决 blocker 时，先集中提问。
- 先读 `AGENTS.md`，再完整读取 PRD 和 Plan，并按改动信号加载相关 `spec/`；检查工作区和全部目标文件，保留已有改动，不暂存或提交。
- 实现前先重述最小改动边界和已确认产品不变量，并用真实调用方核验；现有控制流默认保持不变。按 Plan 的最小范围实现所有步骤，不顺带修复无关缺陷或把局部需求升级为请求级、跨设备或共享模块语义。
- 发现必要联动可以一并处理并记录；但凡需要新增请求级事务、重复项合并、多设备回滚、生命周期锁/refcount、全局 dirty 管理、定向协议封包、重试/迁移框架或修改共享模块行为，均视为实质范围升级，必须先说明触发它的真实可达场景并询问用户，确认前不得实现。
- 执行与风险相称的验证并逐项对应 PRD 验收标准；验证命令严格服从 `AGENTS.md`，禁止项不得放行。若只允许静态核验，则不运行编译、构建、release 或有副作用的 clean，并在 `result.md` 和最终回复中说明未编译及未执行的验证。
- 验证结果必须逐项引用 PRD 中已有的 AC 编号，并为每条 AC 给出对应的源码、调用链或静态断言证据；存在无 AC 编号的验收时停止并请用户先更新 PRD，Implement 不编号或修改 PRD。
- 最后启动只读 `Plan` 子 Agent，不设置 `max_turns`，不修改文件或执行有副作用的命令，只根据实际 diff 和直接调用链审查实施 blocker：某条 AC 未实现或实现结果与 PRD 不符；实际改动偏离 Plan 的最小边界；新增代码存在可从受支持入口触发的局部错误；必要声明、调用方、持久化、注册或通知联动缺失；主 Agent 已完成、拟写入 `result.md` 的静态验证结论没有对应源码证据。审查提示必须写明产品不变量、受支持输入、本次明确不处理的场景和拟记录的静态验证结论。
- 已在 Plan 阶段确认的产品范围、非目标和调用约束不重复评审；只有实际 diff 或新发现的直接调用方证据与 Plan 冲突时，才重新打开范围问题。子 Agent 提出的 blocker 必须给出真实入口到失败点的调用链，并证明没有违反唯一性、数量、顺序或生命周期约束。
- 主 Agent 不得自动实施子 Agent 的建议。先独立复现或静态证明可达，再判断是否属于 PRD；只能由不可能输入、未支持调用方式或范围外系统故障触发的问题应驳回或记录为非 blocker。若建议涉及 Plan 外文件，先判断是否为 AC 必需联动：是则补齐并记录偏差；只有超出 PRD、改变共享行为或明显增加复杂度时才暂停询问用户。
- 每轮修正后比较改动复杂度与原需求；若为修复新引入的机制继续增加事务、锁、回滚或通知状态，优先回退错误假设和过度修复，而不是继续补洞。有效 blocker 修正后重新验证并复审；没有未解决 blocker 即提前结束。
- 完成后判断本次实现是否形成会影响后续 AI 编程决策的稳定、可复用知识，并在 `result.md` 的“偏差与风险”中记录 `Spec 更新建议：需要/无需`及简短理由；Implement 阶段不直接修改 `AGENTS.md` 或 `spec/`。
- 将最终结果写入同目录的 `result.md`；已存在时先完整读取，只保留显式人工备注或能由当前 PRD、Plan 和 diff 重新证明的内容，验收结论、blocker 和终态必须重算。记录实际改动、计划偏差和风险，不修改 `prd.md` 或 `plan.md`。

结果文件结构：`实施结果`、`改动清单`、`验收与验证`、`偏差与风险`、`未解决 blocker`。

完成后回复 PRD、Plan 和 Result 文件路径、终态、评审轮数及剩余 blocker。
