根据以下已落盘 PRD 路径生成最小实施计划：

$@

要求：

- 必须使用参数指定的唯一 `task/<task-name>/prd.md`；当前对话只能补充说明，不能代替落盘 PRD。路径缺失、无效，或不确定项会改变范围和验收时，先集中提问。
- 先读 `AGENTS.md`，按路由加载相关 `spec/`，再核对必要的源码、配置和测试；除最终 `plan.md` 外不修改文件或执行有副作用的命令。
- 先建立最短证据链：`PRD 要求 → 已确认产品不变量 → 真实调用入口/调用方 → 当前行为 → 必要改动`。数组、循环、裸指针、队列或通用 API 的存在，本身不能证明重复输入、请求级事务、并发释放或全局失败语义在产品中可达。
- 把现有实现的业务流程和调用约束视为默认契约；除非 PRD 明确要求改变，优先原地扩展字段、枚举或校验，不重写控制流，不顺带修复无关缺陷，不把局部需求升级为共享基础设施改造。
- 计划必须保持与需求复杂度相称。若一个局部字段/状态调整开始需要请求级事务、跨设备回滚、全局 dirty 管理、新锁/refcount、定向协议封包、重试框架或共享模块行为变化，视为范围升级信号：先重新核验场景是否真实可达；没有明确证据时停止并询问用户，不写入计划。
- 计划先落盘真实入口和产品不变量；每个步骤写明目标、文件或符号、关键联动、保持不变的控制流或错误边界、验证方式及对应 AC。
- 实施步骤和验证方式必须引用 PRD 中已有的 AC 编号，不重新编号、合并或改写验收含义；存在无 AC 编号的验收时停止并请用户先更新 PRD，Plan 不编号或修改 PRD。
- 只列真正影响实施的决策点、风险和 blocker。blocker 必须同时满足：阻止某条明确验收；在已确认产品不变量下可从受支持入口复现；有源码、协议或调用方证据。只能通过违反唯一性、数量、顺序或调用约束构造的场景不是 blocker。
- 最后启动只读 `Plan` 子 Agent，不设置 `max_turns`，不修改文件或执行有副作用的命令，只审查实施前 blocker：某条明确验收无法从已确认的真实入口实现；计划遗漏实现验收必需的直接调用方、持久化、注册、通知或配置联动；实现必须突破当前范围或依赖未确认的产品决策；计划没有为对应验收提供与仓库能力相称的静态核验方式，或计划声称的静态结论超出源码、配置和调用链证据能够支持的范围。审查提示必须列出产品不变量、支持的输入形态和禁止假设的场景，不分析尚不存在的代码缺陷，不构造并发、重复输入、请求级事务、重试或迁移等范围外场景。
- 主 Agent 必须独立核验 blocker 的可达性和范围，不能因子 Agent 给出理论失败序列就自动扩展计划。根据有效证据修订并复审，没有未解决 blocker 即视为达成共识并提前结束；blocker 重复且无新证据或需要扩大范围时停止并询问用户，不伪装成共识。
- 将最终计划写入 PRD 同目录的 `plan.md`。文件已存在时先完整读取；只保留显式人工备注或能由当前 PRD 和证据重新证明的内容，步骤、验收结论和 blocker 必须重算；不修改 `prd.md`。

输出结构：`实施步骤`、`验收与验证`、`决策与风险`、`未解决 blocker`。

完成后回复 PRD 和 Plan 文件路径、评审轮数及剩余 blocker。
